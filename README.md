# WBS プランナー — マルチリリース対応ガントチャート & JIRA 連携ツール

## プロジェクトの計画から JIRA チケット作成まで、ワンストップで

ファイル1つ、インストールなし。開発項目を入力するだけで **WBS 番号付きのガントチャートを自動生成**し、そのまま **JIRA に直接チケットを登録**できます。複数リリースを1枚のガントで管理する**マルチリリース**に対応しています。

[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)
[![No Dependencies](https://img.shields.io/badge/dependencies-none-brightgreen)](wbs-planner.html)
[![Single File](https://img.shields.io/badge/delivery-single%20HTML-orange)](wbs-planner.html)

**[→ wbs-planner.html をダウンロードして開くだけで使えます](wbs-planner.html)**

---

## このツールが解決すること

プロジェクト開始時、「誰が・いつ・何日かけて・何をやるか」を決めるのは手間がかかります。

- Excel でガントチャートを手動で書く → 担当者の空き状況を手で考える
- JIRA にチケットを先に作る → スケジュールが不確かなまま日付を入れる
- Mermaid でガントチャートを書く → 担当者の割り当ては自動化できない

このツールは「**計画フェーズ**」と「**JIRA 登録**」の間のギャップを埋めます。

## ワークフロー

```text
1. リリースを定義する    2. アイテムを入力する    3. 自動スケジュール      4. JIRA に登録する
┌──────────────┐       ┌──────────────────┐    ┌──────────────────┐   ┌──────────────────┐
│ リリースA     │       │ 車両テスト        │    │ WBS番号付き       │   │ Task / Sub-task  │
│ 2026/4〜7/31 │  →    │   要件定義 5日    │ →  │ ガントチャート    │ → │ REST API で直接  │
│ リリースB     │       │   設計開発 10日   │    │ スケジュール表    │   │ 一括作成         │
│ 2026/6〜10/31│       │ ベンチテスト …    │    │                  │   │                  │
└──────────────┘       └──────────────────┘    └──────────────────┘   └──────────────────┘
  複数リリースを1枚          担当者・工数を設定        担当者の空き状況を          Epic・担当者も
  のガントで管理             フェーズごとに制約可       考慮して自動計算            自動設定
```

担当者の空き状況・稼働率・担当可能工程を考慮したスケジュールが自動で決まり、Atlassian Cloud REST API 経由で JIRA に直接登録できます。

## 主な機能

### マルチリリース対応

複数のリリースを1枚のガントチャートで管理できます。

- **リリースごとに独立設定** — 開始日・リリース日・評価ゾーン・識別色・JIRA Epic キー
- **担当者はプロジェクト全体で共有** — リリースをまたいで同一人物に自動割り当て
- **リリース優先度** — リリース配列の先頭を優先してスケジューリング（重複期間でも整合）
- **後方互換** — 旧JSON（単一リリース形式）を読み込むと自動的に「リリース1」に変換

**ガントチャートのレイアウト例：**

```text
ヘッダー行（時間軸）
▼ リリースA  2026/04/01〜07/31  ← 識別色の見出し行
  1 車両テスト
    └ 1.1 要件定義  ████
    └ 1.2 設計開発       ████████
  2 ベンチテスト
    └ 2.1 設計開発              ████            ░░░░░ ← 評価ゾーンA  📦
▼ リリースB  2026/06/01〜10/31  ← 別の識別色の見出し行
  1 レポート作成
    └ 1.1 テストケース作成               ████
    └ 1.2 レポート                            ██  ░░░ ← 評価ゾーンB  📦
```

### WBS 自動付番

アイテムに `1`, `2`, `3`… の番号、フェーズに `1.1`, `1.2`… の枝番を自動で付けます。ガントチャートの行ラベルとスケジュールサマリー表に表示され、JIRA の課題 Summary にもそのまま反映されます。

### JIRA 直接登録（REST API）

「JIRA に登録」ボタン1つで、Atlassian Cloud REST API 経由でチケットを直接作成します。管理者権限は不要で、プロジェクトへの課題作成権限があれば使えます。

アイテムを **Task**、フェーズを **Sub-task** として作成し、リリースごとの Epic への紐付けと担当者の設定も自動で行います。

| フィールド | Task（アイテム） | Sub-task（フェーズ） |
| ---- | ---- | ---- |
| `Issue Type` | `Task` | `Sub-task` |
| `Summary` | `1 コア機能A` | `1.1 要件定義 — コア機能A` |
| `Epic Link` | そのリリースの Epic キー | — |
| `Parent` | — | 親 Task |
| `Assignee` | 最初のフェーズ担当者 | 当該フェーズの担当者 |
| `Description` | カテゴリ・メモ・合計工数 | 担当チーム・稼働日数 |

「**JIRA ユーザー取得**」ボタンでプロジェクトの担当可能ユーザー一覧を取得し、担当者ごとに JIRA アカウントをドロップダウンでマッピングできます。ユーザー検索欄では `/user/picker` API による**ライブ検索**も使えます。

「**フィールド取得**」ボタンでプロジェクトの必須カスタムフィールドを自動取得し、登録前に値を設定できます（担当チーム・承認者など）。

### 自動スケジューリング（LPT アルゴリズム）

残りパイプライン工数が多いタスクを優先的に割り当てる LPT（Longest Processing Time）方式で、全体の完了を最短化します。担当者の空き状況を1営業日単位でシミュレーションします。

```text
優先度 = 自タスク以降のフェーズ工数の合計

例: [要件定義 5日] → [設計開発 20日] → [テスト 3日]
  └ 要件定義の優先度 = 5+20+3 = 28日
  └ より長いパイプラインを持つタスクが先に担当者を確保
```

### リソース制約

| 設定 | 説明 |
| ---- | ---- |
| `allowedPeople` | フェーズを担当できる人を限定（省略時は適任者を自動選択） |
| `requireAll` | 指定した全員が同時着手（ペアプロ・共同作業に） |
| `background` | 工数を消費しない並行進行タスク（調達待ち・外注など） |
| `fixedStart` | 特定日に強制開始（外部依存がある場合） |
| `availableFrom` | 参画開始日（中途合流メンバーに） |
| `utilization` | 稼働率（0.8 = 80%稼働） |

### 営業日カレンダー

- 土日を自動除外
- 国民の祝日をバンドル済み（2024〜2028年）
- [holidays-jp API](https://holidays-jp.github.io) からワンクリックで最新データに更新
- 会社独自の休業日（年末年始・創立記念日など）を別途登録可能

### 表示・エクスポート

- **3段階ズーム**: 日 / 週 / 月 単位を切替（3ヶ月プロジェクトは週単位、1年超は月単位など）
- **リアルタイムプレビュー**: 設定変更のたびに即時反映
- **HTML 出力**: スタンドアロン HTML として保存・共有
- **PNG 出力**: スライド・ドキュメントに貼り付け可能な画像として保存
- **JSON 保存 / 読込**: 設定ファイルをチームで共有・バージョン管理

## クイックスタート

### 1. ファイルを開く

[`wbs-planner.html`](wbs-planner.html) をダウンロードし、ブラウザで開きます。

> **JIRA 連携を使う場合**は、以下の2コマンドを別々のターミナルで起動します。
>
> ```bash
> # ターミナル 1: HTML ファイルを配信
> python3 -m http.server 8000
>
> # ターミナル 2: JIRA CORS プロキシを起動
> python3 scripts/jira-proxy.py
> ```
>
> ブラウザで `http://localhost:8000/wbs-planner.html` を開き、JIRA タブの「CORS プロキシ URL」に `http://localhost:8001` を設定してください。

### 2. 工程タイプを定義する

「**工程タイプ**」タブで、プロジェクトで使う工程の種類・担当チーム・色を設定します。

### 3. 担当者を登録する

「**担当者**」タブで担当者ごとに担当可能な工程・参画開始日・稼働率を設定します。

### 4. リリースとアイテムを入力する

「**リリース**」タブでリリースを追加し、各リリースの中にアイテムとフェーズを設定します。  
WBS 番号（`1`, `1.1` など）は自動で付番されます。

### 5. プロジェクト設定を確認する

「**プロジェクト設定**」タブでプロジェクト名・デフォルト開始日・ガント表示単位を設定します。

### 6. JIRA に登録する

「**JIRA**」タブを開き、以下の順で操作します。

1. **接続設定**にサイト URL・メールアドレス・API トークン・プロジェクトキーを入力
2. 「接続テスト」で認証を確認
3. 「JIRA ユーザー取得」でプロジェクトの担当者一覧を取得し、各担当者と JIRA アカウントをマッピング
4. 「フィールド取得」で必須カスタムフィールドを取得して値を設定
5. 「**JIRA に登録**」で Task / Sub-task を一括作成

> API トークンは [Atlassian アカウント設定](https://id.atlassian.com/manage-profile/security/api-tokens) の「Security → API tokens」から発行できます。  
> 設定は `localStorage` に自動保存されます。認証情報（API トークン）は JSON エクスポートに含まれません。  
> チームで共有する場合は「JSON 保存」でファイルに書き出してください。

## 詳細な操作手順

### 工程タイプの設定

工程タイプはフェーズの種類を定義します。プロジェクト固有の工程を追加してください。

1. 「**工程タイプ**」タブを開く
2. 「**＋ 追加**」ボタンをクリック → 「新工程」が追加される
3. 追加されたカードをクリックして展開し、以下を設定:
   - **工程名**: 例 `要件定義`、`設計開発`、`テストケース作成`
   - **チーム**: 担当チーム名（JIRA の Sub-task Description に記載される）
   - **カラー**: ガントチャートのバーの色
4. 不要な工程は「**削除**」ボタンで削除

> **注意**: 工程名を変更すると、その工程を使っているフェーズの表示も更新されます。

---

### 担当者の設定

1. 「**担当者**」タブを開く
2. 「**＋ 追加**」ボタンをクリック
3. カードを展開して設定:
   - **名前**: 担当者の名前（JIRA マッピング時に使用）
   - **チーム**: 所属チーム
   - **担当可能工程**: この担当者が担当できる工程タイプをチェック（複数選択可）
   - **参画開始日**: 途中参画の場合に設定（空欄 = プロジェクト開始から参画）
   - **稼働率**: `100%` = フルタイム、`80%` = 他プロジェクト兼務など
   - **備考**: 任意のメモ

> **稼働率について**: 現在のバージョンでは担当者の割り当て判定（空きかどうか）のみに使用しており、1タスクあたりの期間は `days` フィールドで直接指定します。

---

### リリースの追加と設定

1. 「**リリース**」タブを開く
2. 「**＋ リリース追加**」ボタンをクリック → 新しいリリースが追加される
3. リリースカードをクリックして展開し、以下を設定:

| 設定項目 | 説明 |
| ---- | ---- |
| **リリース名** | 例 `リリースA`、`v1.0`、`フェーズ1` |
| **識別色** | ガントのリリース見出し行と評価ゾーンに使う色 |
| **開始日** | このリリースのアイテムのスケジューリング開始日 |
| **リリース日** | 締め切り日（ガントに 📦 マーク） |
| **リリース評価ゾーン** | ガントに評価期間のオーバーレイを表示するかどうか |
| **評価期間** | リリース日の何日/週/ヶ月前からが評価期間か |
| **JIRA Epic キー** | このリリースのアイテムを登録する Epic（例: `PROJ-1`） |

複数のリリースが重なる期間でも、担当者はプロジェクト全体で自動的に共有・調整されます。

---

### アイテムとフェーズの設定

各リリースの中にアイテム（開発項目）を追加し、フェーズを設定します。

#### アイテムの追加

1. リリースカードを展開する
2. 「**＋ アイテム追加**」ボタンをクリック
3. アイテムカードを展開して設定:
   - **アイテム名**: 例 `車両テスト`、`コア機能A`
   - **カテゴリ**: グループ化用ラベル（ガントのバッジに表示）
   - **メモ**: 任意の補足説明

#### フェーズの追加

1. アイテムカードを展開する
2. 「**＋ フェーズ**」ボタンをクリック
3. フェーズ行を設定:
   - **工程タイプ**: ドロップダウンから選択
   - **日数**: 営業日数（工数）

#### 担当者制約（オプション）

| 設定 | 使い方 |
| ---- | ---- |
| **許可担当者** | チェックすると、そのフェーズに担当できる人を限定できます（未チェック = 担当可能な工程を持つ人を自動選択） |
| **全員同時着手** | 許可担当者全員が同じ日に開始する必要がある場合（ペアプロ・合同レビューなど） |
| **バックグラウンド** | 担当者の工数を消費しない並行タスク（外注・部品調達待ちなど） |
| **固定開始日を設定** | 外部依存がある場合に特定日から強制開始 |

---

### プロジェクト設定

「**プロジェクト設定**」タブで以下を設定します。

| 設定項目 | 説明 |
| ---- | ---- |
| **プロジェクト名** | ガントチャートのタイトル・JSON ファイル名に使用 |
| **デフォルト開始日** | 新規リリース追加時の開始日の初期値 |
| **ガントチャート表示単位** | 日 / 週 / 月 を切替 |
| **祝日（国民の祝日）** | バンドル済み（2024〜2028年）。「自動取得」で最新に更新可能 |
| **会社指定休業日** | 年末年始・創立記念日など会社独自の休業日を追加 |
| **プロジェクトをリセット** | 全設定を初期状態に戻す（取り消し不可） |

### JIRA 連携の手順

#### 事前準備

JIRA の CORS 制限を回避するため、付属のプロキシスクリプトを使用します。

```bash
# ターミナル 1: HTML ファイルを配信（ファイル直接開きの場合は不要）
python3 -m http.server 8000

# ターミナル 2: CORS プロキシを起動
python3 scripts/jira-proxy.py
# → "JIRA proxy listening on port 8001" と表示されれば起動成功
```

#### 接続設定

「**JIRA**」タブを開き、接続設定を入力します。

| 項目 | 入力例 | 説明 |
| ---- | ---- | ---- |
| **JIRA サイト URL** | `https://mycompany.atlassian.net` | `.atlassian.net` まで含めて入力 |
| **メールアドレス** | `user@example.com` | Atlassian アカウントのメール |
| **API トークン** | `ATATxxxx…` | 下記リンクから発行 |
| **プロジェクトキー** | `PROJ` | JIRA プロジェクトのキー（大文字） |
| **CORS プロキシ URL** | `http://localhost:8001` | プロキシを起動した場合のみ |

> API トークンは [Atlassian アカウント設定 → Security → API tokens](https://id.atlassian.com/manage-profile/security/api-tokens) から発行できます。

「**接続テスト**」ボタンをクリックし、「接続成功」と表示されれば OK です。

#### 担当者マッピング

1. 「**JIRA ユーザー取得**」をクリック → プロジェクトの担当可能ユーザーを取得
2. 各担当者の横のドロップダウンから対応する JIRA アカウントを選択
3. ドロップダウンに出ない場合は検索欄に名前を入力してライブ検索

#### 必須フィールドの設定

プロジェクトによっては Task 作成時に必須カスタムフィールド（担当チーム・承認者など）が必要な場合があります。

1. 「**フィールド取得**」をクリック → 必須フィールドを自動取得
2. 課題タイプのドロップダウンから Task に相当するタイプを選択（`Task` または `タスク` が自動選択される）
3. 表示されたフィールドに値を設定
4. ユーザー型フィールドは検索欄に名前を入力してライブ検索

#### リリースごとの Epic キー設定

各リリースに登録先の JIRA Epic を設定します。

1. 「**リリース**」タブを開く
2. リリースカードを展開
3. 「**JIRA Epic キー**」欄に Epic のキー（例: `PROJ-100`）を入力

Epic キーが設定されているリリースのアイテムのみ、Epic に紐付けられて登録されます。

#### JIRA への登録実行

「**JIRA に登録**」ボタンをクリックすると、全リリースのアイテムを順番に登録します。

- リリースA のアイテムを Task + Sub-task として登録
- リリースB のアイテムを Task + Sub-task として登録
- ログ欄にリアルタイムで進捗が表示される

### データの保存と共有

#### 自動保存

全設定は **localStorage**（ブラウザ内）に自動保存されます。ページを閉じても設定は保持されます。

#### JSON として保存・共有

チームメンバーと設定を共有したい場合は JSON ファイルを使います。

- **JSON 保存**: ページ下部の「JSON 保存」ボタンをクリック → `プロジェクト名-config.json` がダウンロードされる
- **JSON 読込**: 「JSON 読込」ボタンをクリック → 保存した JSON を選択して復元

> **注意**: API トークン等の認証情報は JSON ファイルに含まれません。チーム共有時も安全です。

#### HTML / PNG エクスポート

- **HTML 出力**: 現在のガントチャートをスタンドアロン HTML として保存（JavaScript 不要で閲覧可能）
- **PNG 出力**: ガントチャートを画像として保存（スライドへの貼り付けに）

## 設定ファイル（JSON）の形式

```jsonc
{
  "projectName": "プロジェクト名",
  "startDate": "2026-04-20",      // 新規リリース追加時のデフォルト開始日
  "ganttUnit": "weeks",           // "days" | "weeks" | "months"

  "holidays": {
    "national": ["2026-04-29", "2026-05-03"],  // 国民の祝日
    "company":  ["2026-12-29", "2026-12-30"]   // 会社指定休業日
  },

  "phaseTypes": [
    { "name": "要件定義", "team": "企画", "color": "#3B82F6" },
    { "name": "設計開発", "team": "開発", "color": "#10B981" }
  ],

  "people": [
    {
      "name": "Alice",
      "team": "企画",
      "phases": ["要件定義"],         // 担当可能な工程タイプ
      "availableFrom": null,          // null = プロジェクト開始から参画
      "utilization": 1.0,             // 稼働率（0.5 = 50%稼働）
      "note": "",
      "jiraUser": ""                  // JIRA アカウント ID（JIRA タブで自動設定）
    },
    {
      "name": "Bob",
      "team": "開発",
      "phases": ["設計開発"],
      "availableFrom": "2026-05-01",  // 5月から参画
      "utilization": 0.8,             // 80%稼働
      "note": "兼務あり",
      "jiraUser": ""
    }
  ],

  // リリース配列（各リリースに開発アイテムを内包）
  "releases": [
    {
      "id": "r_abc123",              // 内部ID（自動生成）
      "name": "リリースA",
```

## データの取り扱いとプライバシー

**プロジェクト情報（アイテム名・担当者名・スケジュール等）は外部に送信されません。**

| タイミング | 接続先 | 送受信されるデータ |
| ---- | ---- | ---- |
| 通常操作（常時） | なし | — |
| 「祝日自動取得」ボタンを押したとき | [holidays-jp.github.io](https://holidays-jp.github.io) | 祝日一覧の受信のみ（プロジェクトデータは送らない） |
| 「PNG 出力」ボタンを押したとき | [cdnjs.cloudflare.com](https://cdnjs.cloudflare.com)（html2canvas） | ライブラリの受信のみ |
| 「JIRA に登録」ボタンを押したとき | ユーザーが設定した自社 JIRA サイト | 登録する課題データ（意図的な送信） |

データは `localStorage`（ブラウザ内）にのみ保存され、サーバーへのアップロードは一切行われません。JIRA の API トークン等の認証情報もブラウザ外には出ません。

社内プロジェクト名・担当者名・スケジュール情報を入力しても問題ありません（JIRA 登録時を除いて外部に送信されないため）。

## Mermaid・PlantUML との違い

Mermaid や PlantUML のガントチャートは「すでに決まったスケジュールを図にする」ツールです。このツールは「**誰が・いつ・何日かけてやるか**を自動で決める」ところから始めます。

### 機能比較

| 機能 | このツール | Mermaid | PlantUML |
| ---- | :---------: | :-------: | :--------: |
| インストール不要 | ✅ | ✅ (CDN利用時) | ❌ Java必須 |
| GUI エディタ | ✅ | ❌ コード記述 | ❌ コード記述 |
| **マルチリリース管理** | ✅ | ❌ | ❌ |
| **担当者の自動割り当て** | ✅ | ❌ | ❌ |
| **リソース競合の自動回避** | ✅ | ❌ | ❌ |
| **担当可能工程の制約** | ✅ | ❌ | ❌ |
| **稼働率・参画開始日の考慮** | ✅ | ❌ | ❌ |
| **並行バックグラウンドタスク** | ✅ | ❌ | ❌ |
| **営業日カレンダー（祝日対応）** | ✅ | ❌ | ❌ |
| **リリース評価超過の警告** | ✅ | ❌ | ❌ |
| **WBS 番号自動付番** | ✅ | ❌ | ❌ |
| **JIRA 直接登録（REST API）** | ✅ | ❌ | ❌ |
| リアルタイムプレビュー | ✅ | △ (要ビルド) | ❌ |
| 日 / 週 / 月 ズーム切替 | ✅ | ❌ | ❌ |
| PNG / HTML エクスポート | ✅ | △ | △ |
| JSON で設定保存・復元 | ✅ | △ (コード自体) | △ (コード自体) |
| テキストベースでバージョン管理 | △ (JSON) | ✅ | ✅ |

## 技術仕様

| 項目 | 内容 |
| ---- | ---- |
| 配布形式 | 単一 HTML ファイル（約80KB） |
| 外部依存 | なし（PNG出力時のみ html2canvas を CDN から遅延ロード） |
| 動作環境 | モダンブラウザ（Chrome / Firefox / Safari / Edge） |
| データ保存 | localStorage（ブラウザ内） |
| スケジューリング | LPT アルゴリズム、1営業日単位シミュレーション、マルチリリース共有担当者プール |
| 祝日データ | 2024〜2028年バンドル済み + [holidays-jp API](https://holidays-jp.github.io) 対応 |
| JIRA 連携 | Atlassian Cloud REST API v3 経由で Task / Sub-task を直接作成（管理者権限不要） |
